# Example: Python Project after `/setup-repo`

This shows the state of a Python project after running `/setup-repo` with `claude-onboarding-kit` installed.

---

## Directory Structure

```text
my-python-app/
├── .claude/
│   ├── skills/                    # Symlinks to ~/.claude/kit/skills/
│   │   ├── setup-repo -> ...
│   │   ├── create-prd -> ...
│   │   └── create-adr -> ...
│   └── hooks/
│       └── session-start.sh       # Auto-generated from session-start-python.sh template
├── .github/
│   ├── workflows/
│   │   ├── security.yml           # CodeQL + pip-audit scanning
│   │   └── release.yml            # Release workflow
│   └── pull_request_template.md
├── src/
│   └── my_python_app/
│       ├── __init__.py
│       └── main.py
├── tests/
│   ├── __init__.py
│   └── test_main.py
├── .venv/                         # Virtual environment (git-ignored)
├── .ruff.toml                     # Generated by lint-install (ruff plugin)
├── mypy.ini                       # Generated by lint-install (mypy plugin)
├── .gitleaks.toml                 # Allowlist starter for secrets scanning
├── hookify.bare-except.local.md   # Hookify rule: block bare exception swallowing
├── hookify.secrets.local.md       # Hookify rule: block hardcoded secrets
├── CLAUDE.md                      # Project instructions for Claude Code
├── AGENTS.md                      # Multi-agent context file
├── Makefile                       # Generated lint/test/format targets
├── pyproject.toml                 # Project metadata, dependencies, tool config
└── pytest.ini                     # Pytest configuration (or in pyproject.toml)
```

## Commands That Work After Setup

```bash
# Virtual environment setup (one time)
python3 -m venv .venv
.venv/bin/python -m pip install -e ".[dev]"

# Linting
make lint                      # Run all lint targets (ruff, mypy, shellcheck)
ruff check src/                # Ruff lint
ruff format src/               # Ruff format (in-place)
mypy src/                      # Type checking

# Testing
pytest --timeout=120 -x -q     # Standard run: stop on first failure, quiet output
pytest -n 6                    # Parallel run (6 workers, for heavy suites)
pytest -n auto                 # Parallel run (auto-detect workers)
pytest -n 0                    # Sequential run (use for debugging)
pytest --cov=src tests/        # With coverage report

# Code quality (kit scripts)
ollama-code-review src/        # AI code review via local Ollama model
generate-embeddings src/       # Generate semantic search embeddings

# Security
pip-audit                      # Python dependency audit
# Security workflow runs on every push via .github/workflows/security.yml

# Lessons / anti-patterns
lesson-check --project-root .          # Check for known anti-patterns
lessons-db scan --target . --baseline HEAD             # Check only staged files (pre-commit use)

# Project skills (invoke via Claude Code)
/setup-repo                    # Re-run or update pipeline phases
/create-prd <feature>          # Generate a Product Requirements Document
/create-adr <decision>         # Record an Architecture Decision Record
/create-tech-spec <topic>      # Generate a technical specification
```

## Notes

- `session-start.sh` runs automatically at the start of each Claude Code session. It checks memory, reads `progress.txt`, and surfaces any blocked lessons.
- Use `.venv/bin/python -m pip install` (not `.venv/bin/pip`) — Homebrew PATH can corrupt pip behavior when Homebrew Python is first on PATH.
- `pyproject.toml` is the single source of truth for project metadata, dependencies, and tool configuration (ruff, mypy, pytest settings).
- `hookify.bare-except.local.md` blocks bare `except:` clauses at write-time — this is the highest-impact lesson (#7) in the kit's default ruleset.
- Re-running `/setup-repo` is safe — all scripts are idempotent.
